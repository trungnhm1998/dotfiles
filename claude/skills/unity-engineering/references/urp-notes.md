## 1. SRP Batcher

The SRP Batcher reduces CPU overhead by making material constant buffer data persistent on the GPU across draw calls. Instead of re-uploading all material properties before every draw, Unity only re-uploads what changed.

**What keeps materials batched:**
- Objects sharing the **same shader variant** can be batched together (they don't need to share a material asset).
- All material properties must be declared inside a `CBUFFER_START(UnityPerMaterial) / CBUFFER_END` block in the shader. Any property outside this block breaks compatibility.
- Built-in URP shaders (Lit, Unlit, Simple Lit) are SRP Batcher compatible by default.

**What breaks SRP Batcher compatibility:**
- Custom shaders that declare properties outside `UnityPerMaterial` CBUFFER.
- `MaterialPropertyBlock` — per-object property overrides bypass the SRP Batcher. Consider using GPU instancing or texture arrays instead if you need per-object color/data.
- GPU instancing being enabled on a material (the two systems are mutually exclusive — enabling instancing disables SRP Batching for that material).

**Diagnosing breaks:** Open Frame Debugger (Window > Analysis > Frame Debugger), expand the render pass, and click an SRP Batch. It shows exactly why a draw call wasn't merged with the previous one (e.g., "Node has different shader", "MaterialPropertyBlock is used").

**Per-material vs per-object data:**
- `UnityPerMaterial` CBUFFER → per-material data, persistent in GPU memory across draw calls with the same shader.
- `UnityPerDraw` CBUFFER (built-in) → per-object data (transform matrices, etc.), updated via a dedicated code path each draw call. This is why different meshes with the same shader can still be SRP-batched.

## 2. GPU Instancing vs SRP Batcher

| | **SRP Batcher** | **GPU Instancing** |
|---|---|---|
| Draw call reduction | CPU setup reduction (not fewer DC) | Merges multiple draws into one |
| Requirement | Same shader variant | Same mesh + same material |
| Per-object variation | Via `UnityPerDraw` (transforms only by default) | Via instanced properties (`UNITY_INSTANCING_BUFFER`) |
| Compatible together | No — mutually exclusive per material | — |
| Best for | General scene objects with varied meshes | Many identical meshes (grass, trees, crowd) |

**When to use GPU instancing:**
- Hundreds of instances of the same mesh + material (e.g., foliage, projectiles, particles rendered as meshes).
- When you need per-instance color/data variation on identical meshes.
- Enable via: Material Inspector > Advanced > Enable GPU Instancing. Note this disables SRP Batching for that material.

**Rule of thumb:** SRP Batcher is the right default for most scene objects. Switch to GPU instancing only for high-count identical-mesh scenarios, and verify the switch actually helps with the Profiler.

## 3. Renderer Features & Passes

A **Renderer Feature** is a modular extension point in the URP Renderer asset. It injects one or more `ScriptableRenderPass` instances into the frame at a specified `RenderPassEvent`.

**When to add a custom Renderer Feature:**
- Post-processing effects not covered by URP's Volume system (e.g., custom outline pass, portal rendering, screen-space decals).
- Stencil-based masking passes.
- Extra geometry passes (e.g., render only a specific layer with a different shader).

**Unity 6 note:** Custom Renderer Features in Unity 6 use the **RenderGraph API** (`RecordRenderGraph` / `RasterGraphContext`). The legacy `Execute(ScriptableRenderContext, ref RenderingData)` path still compiles but is deprecated — new features should use RenderGraph for forward compatibility.

```csharp
// Unity 6 RenderGraph Renderer Feature skeleton (see also template generated by
// Assets > Create > Rendering > URP Renderer Feature)
public override void RecordRenderGraph(RenderGraph renderGraph, ContextContainer frameData)
{
    using var builder = renderGraph.AddRasterRenderPass<PassData>("MyPass", out var passData);
    var resourceData = frameData.Get<UniversalResourceData>();
    builder.SetRenderAttachment(resourceData.activeColorTexture, 0);
    builder.SetRenderFunc((PassData data, RasterGraphContext ctx) => ExecutePass(data, ctx));
}
```

**Common injection points:**
- `RenderPassEvent.AfterRenderingOpaques` — scene color/depth available, transparents not yet drawn.
- `RenderPassEvent.BeforeRenderingTransparencies` — after opaques, before transparents.
- `RenderPassEvent.AfterRenderingPostProcessing` — full post stack applied, good for UI-adjacent effects.

**Performance caution:** Requesting a copy of the color or depth buffer (`ConfigureInput(ScriptableRenderPassInput.Color | Depth)`) forces a blit and breaks TBDR efficiency on mobile. Only request what you actually need.

## 4. URP Asset & Quality Settings

These are the settings most likely to affect a solo dev's performance. Access via Project Settings > Quality (or Graphics > URP Asset).

| Setting | Recommendation | Why |
|---|---|---|
| **Rendering Path** | Forward (default) for mobile/2D; Deferred for desktop 3D with many lights | Forward is cheaper per-pixel with few lights |
| **HDR** | Disable on mobile / low-end; 32-bit if enabled on PC | Bloom and tonemapping require HDR; adds memory bandwidth cost |
| **Anti-aliasing (MSAA)** | 2x or Off for mobile; 4x for desktop | Significant memory bandwidth cost, especially on TBDR GPUs |
| **Render Scale** | 1.0 default; reduce (e.g., 0.75) for GPU-bound scenes | Scales the render target resolution, major GPU win |
| **Shadow Distance** | Minimize to what's visible | Shadow maps are expensive; culling distant shadows saves GPU |
| **Additional Lights** | Per-Vertex or Disabled for mobile; Per-Pixel for desktop | Per-pixel point lights add a pass per light in Forward |
| **Soft Shadows** | Disable on mobile | Expensive PCF filter |
| **Depth Texture** | Disable unless a shader/effect needs it | Unnecessary fullscreen blit |
| **Opaque Texture** | Disable unless needed for refraction | Another fullscreen blit |
| **Volume Update Mode** | "Via Scripting" for shipped builds | Prevents Unity from re-evaluating all Volume components every frame |

**Multiple URP Assets:** Create a separate URP asset per quality level (Low/Medium/High) and assign them in Project Settings > Quality. Lets mobile and desktop configs diverge without code changes.

## 5. 2D Renderer vs 3D

URP has two renderer types for a Renderer asset: **Universal Renderer** (3D / 2D hybrid) and the legacy **2D Renderer** (now called "2D Renderer" in the URP asset dropdown).

**When to use the 2D Renderer:**
- Pure 2D games using Sprite Renderer and Tilemap Renderer.
- Enables 2D Lights (point, global, spot, freeform), Normal Map on sprites, and 2D shadow casters.
- 2D-specific post processing (Pixel Perfect Camera).

**Setup differences:**
- In the URP asset, set Renderer Type to "2D Renderer".
- Sprites use `Universal 2D` shader variants (e.g., `Sprites/Lit` or `Sprites/Unlit`).
- 2D Lights are `Light2D` components; set the `Target Sorting Layers` on each light to control which sprites it illuminates.
- The 2D Renderer does NOT support most 3D Renderer Features — don't mix the two for a pure 2D game.

**2D Lights basics:**
- **Global Light 2D** — ambient fill for the whole scene, applied to all target layers.
- **Point Light 2D** — radial falloff from a point; uses a falloff texture.
- **Freeform Light 2D** — arbitrary polygon shape, good for area lights.
- Normal maps on sprites: set the sprite's Normal Map in the Sprite Atlas or as a secondary texture in the Sprite asset, then set the material to use `Sprites/Lit`.

**SRP Batcher note for 2D:** The 2D Renderer has its own batching behavior. `Sprite Renderer` components with the same material and no `MaterialPropertyBlock` can batch, but the SRP Batcher operates differently for 2D — use the Frame Debugger to verify batching in your specific setup.

## 6. Common URP Gotchas

### Built-in → URP Shader Incompatibility
Standard Shader (`Standard`, `Legacy Shaders/*`) does not work in URP — objects appear pink. Fix: reassign materials to URP equivalents (`Universal Render Pipeline/Lit`, `Universal Render Pipeline/Unlit`). Use the automatic upgrader at Edit > Rendering > Materials > Convert All Built-in Materials to URP as a starting point, then manually verify complex materials.

### Transparency Sorting (2D)
Transparent objects in 3D URP are sorted by distance from camera. In 2D, they're sorted by `Sorting Layer` and `Order in Layer`. If mixing 3D objects and 2D sprites, set the Camera's **Transparency Sort Mode** to "Custom Axis" (Y-axis for top-down games) in the Camera component.

### Volume-Based Post-Processing
URP post-processing is controlled by `Volume` components, not the Camera's legacy image effect pipeline. Add a `Volume` component to a GameObject (or use a Global Volume), create a `Volume Profile`, and add overrides (Bloom, Color Grading, Depth of Field, etc.). The camera must have a `Volume Mask` layer set to the layer where your volumes live. **Gotcha:** All cameras need `Post Processing` enabled in the Camera component — it's off by default.

### Shader Graph and SRP Batcher
Shader Graph-generated shaders are SRP Batcher compatible as long as you don't route non-per-material properties through a `Custom Function` node that reads from outside `UnityPerMaterial`. If a Shader Graph shader breaks batching, check the Frame Debugger message — it usually tells you which property is the culprit.

### Missing Post-Processing Effects
URP does not include all effects from the Post Processing Stack v2 (used in Built-in). Lens Dirt, Screen Space Ambient Occlusion (available in URP 12+), and some advanced effects require custom Renderer Features or third-party packages.
